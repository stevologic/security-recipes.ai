{{< callout type="info" >}}
**What this page documents.** This is the MCP server that actually ships
in this repository. It is a read-only FastMCP server backed by the
generated SecurityRecipes site index and local generated JSON packs. It is
not a catalog of live Snyk, Jira, GitHub, Slack, observability, or cloud
posture connectors.
{{< /callout >}}

SecurityRecipes ships a standalone MCP server for agents that need
structured access to the open recipe corpus and the generated control,
policy, evidence, trust, red-team, readiness, and identity packs in this
repo.

The implementation lives in `mcp_server.py`. The default configuration
template is `mcp-server.toml.example`. The container image is defined by
`Dockerfile.mcp-server`, and the localhost connection notes live in
`README.mcp-localhost.md`.

## What Ships

The repo-shipped server is:

- **Read-only.** It searches and returns SecurityRecipes content and
  generated JSON packs. It does not create PRs, edit tickets, rotate
  secrets, run scanners, or write to external systems.
- **FastMCP-based.** The server uses `fastmcp==3.2.0` and registers the
  MCP server as `security-recipes-mcp`.
- **Index-backed.** Recipe search and retrieval read from the Hugo
  `recipes-index.json` generated by `layouts/index.recipesindex.json`.
- **Pack-backed.** Governance tools read local files under
  `data/control-plane`, `data/policy`, and `data/evidence`.
- **Configurable.** `mcp-server.toml.example` lets a fork point the
  server at a custom site index and custom pack paths without changing
  Python code.
- **HTTP-ready.** Local Python defaults to Streamable HTTP on
  `http://0.0.0.0:8000/mcp`. The Docker image listens on port `80` at
  `/mcp`, which is normally mapped to a host port such as `8123`.

The server can also run as `stdio`, `http`, `streamable-http`, or `sse`
by setting `RECIPES_MCP_TRANSPORT`.

## Data Model

The MCP server has two data sources.

First, it fetches `source_index_url`, which defaults to:

```text
https://security-recipes.ai/recipes-index.json
```

That index is the generated search surface for regular Hugo pages. Each
record includes fields such as `slug`, `title`, `url`, `path`, `section`,
`agent`, `tags`, `severity`, `summary`, `content`, `last_updated`, and
`source_file`.

Second, it reads local generated packs from the filesystem:

`recipes_mcp_connector_intake_pack` returns pre-promotion decisions for
candidate MCP servers. Neither tool connects to upstream systems.

## Runtime Behavior

At startup, `mcp_server.py` loads configuration and performs an eager
refresh of the recipe index. That startup fetch is intentional: a bad
index URL, missing hostname, disallowed hostname, invalid JSON payload, or
missing required index fields fails early instead of leaving agents with a
half-working tool surface.

Recipe index reads are cached in memory. The relevant controls are:

- `cache_ttl_seconds`, default `3600`.
- `request_timeout_seconds`, default `15`.
- `max_results_default`, default `8`.
- `max_results_cap`, default `25`.

The server sends `If-None-Match` when it has an ETag and the refresh is
not forced. Use `recipes_refresh(force=true)` when an agent host or
operator needs to force a re-fetch.

Runtime environment variables control process behavior:

| Variable | Purpose |
| --- | --- |
| `RECIPES_MCP_CONFIG` | Config file path. Defaults to `./mcp-server.toml`. |
| `RECIPES_MCP_TRANSPORT` | One of `stdio`, `http`, `streamable-http`, or `sse`. Defaults to `streamable-http`. |
| `RECIPES_MCP_HOST` | Bind host for HTTP transports. Defaults to `0.0.0.0`. |
| `RECIPES_MCP_PORT` | Bind port for HTTP transports. Defaults to `8000` outside Docker. |
| `RECIPES_MCP_PATH` | HTTP path. Defaults to `/mcp`, or `/sse` for SSE transport. |
| `RECIPES_MCP_LOG_LEVEL` | Optional FastMCP log level. |

The pack paths also have environment variable defaults in `mcp_server.py`
such as `RECIPES_MCP_GATEWAY_POLICY_PATH`,
`RECIPES_MCP_ASSURANCE_PACK_PATH`,
`RECIPES_MCP_AUTHORIZATION_CONFORMANCE_PACK_PATH`,
`RECIPES_MCP_READINESS_SCORECARD_PATH`, and
`RECIPES_MCP_CONTEXT_EGRESS_BOUNDARY_PACK_PATH`,
`RECIPES_MCP_SECURE_CONTEXT_ATTESTATION_PACK_PATH`,
`RECIPES_MCP_SECURE_CONTEXT_EVAL_PACK_PATH`,
`RECIPES_MCP_CONTROL_PLANE_BLUEPRINT_PATH`, and
`RECIPES_MCP_MEASUREMENT_PROBE_PACK_PATH`. The TOML config can
override those resolved defaults.

## Run Locally

From the repository root on Windows PowerShell:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements-mcp-server.txt
Copy-Item mcp-server.toml.example mcp-server.toml
python mcp_server.py
```

The default local endpoint is:

```text
http://localhost:8000/mcp
```

For stdio-only clients, set:

```powershell
$env:RECIPES_MCP_TRANSPORT = "stdio"
python mcp_server.py
```

## Run With Docker

Build the standalone MCP server image:

```powershell
docker build -f Dockerfile.mcp-server -t mcp.server .
```

Run it on localhost:

```powershell
docker run --rm -it -p 8123:80 mcp.server
```

The container sets:

```text
RECIPES_MCP_CONFIG=/app/mcp-server.toml
RECIPES_MCP_TRANSPORT=streamable-http
RECIPES_MCP_HOST=0.0.0.0
RECIPES_MCP_PORT=80
RECIPES_MCP_PATH=/mcp
```

Connect MCP clients to:

```text
http://localhost:8123/mcp
```

To mount a custom config:

```powershell
docker run --rm -it -p 8123:80 `
  -v "${PWD}/mcp-server.toml:/app/mcp-server.toml:ro" `
  mcp.server
```

The root `docker-compose.yml` currently builds the Hugo site container on
port `8080`; it does not start this MCP server. Use the standalone Docker
commands above, or add a separate compose service if you want both the
site and MCP server running together.

## Client Configuration

For MCP clients that accept server JSON:

{
  "mcpServers": {
    "security-recipes": {
      "transport": "streamable-http",
      "url": "http://localhost:8123/mcp"

## How Agents Should Use It

A typical read-only flow looks like this:

1. Call `recipes_match_finding` with a CVE, package, ecosystem, scanner
   rule ID, or keywords from the finding.
2. Call `recipes_get` on the best matching slug or path to retrieve the
   full recipe.
3. Call `recipes_workflow_control_plane`,
   `recipes_mcp_gateway_policy`, or the assurance, identity, trust,
   red-team, and readiness tools when the agent needs workflow controls
   and evidence.
4. Call `recipes_evaluate_mcp_gateway_decision` before a separate
   gateway or orchestrator allows a proposed tool call against enterprise
   systems.

That last point matters: the shipped server can evaluate policy, but it
is not itself a broker in front of other MCP servers.

## Security Boundaries

The shipped server deliberately keeps a narrow boundary:

- It exposes typed MCP tools, not a generic filesystem, SQL, shell, or
  HTTP fetch tool.
- It checks that the configured source index hostname is present in
  `allowed_source_hosts` before fetching the recipe index.
- It reads generated local JSON packs from configured paths.
- It does not store credentials for upstream security tools.
- It does not enforce authentication, authorization, metering, or
  premium entitlement by itself.
- It does not produce audit logs beyond normal process and deployment
  logging.
- It does not rate-limit individual agents or tools beyond the index
  request timeout and result caps.

If you expose this server outside a local development environment, put it
behind the controls your deployment requires: reverse proxy auth, network
allow-lists, TLS, request logging, rate limiting, and any premium access
checks for hosted content or production endpoints.

## Free vs Premium MCP Access

SecurityRecipes remains open by default. The code in this repository
ships the free/open reference MCP server: public recipe retrieval,
community prompt and handbook context through the site index, and local
generated governance packs.

The current premium positioning is a production MCP path for hosted or
enterprise use: agent-verified premium context, premium prompt packs,
higher-rate production endpoints, support/SLA, and enterprise policy
packaging where those are separately provisioned. Those premium
capabilities are not feature flags hidden inside `mcp_server.py`.

| Capability | Repo-shipped free/open server | Premium production MCP positioning |
| --- | --- | --- |
| Public recipes and handbook content | Available through `recipes_search`, `recipes_list`, and `recipes_get` | Available |
| Community prompt context | Available when present in the generated site index | Available |
| Generated control, policy, evidence, trust, red-team, readiness, and identity packs | Available from local JSON files in this repo | Available, with hosted/enterprise packaging where provisioned |
| Runtime gateway-policy decision evaluation | Available through `recipes_evaluate_mcp_gateway_decision` | Available as part of a production control path where provisioned |
| Runtime MCP authorization conformance evaluation | Available through `recipes_evaluate_mcp_authorization_decision` | Available as hosted auth discovery, metadata checks, and scope-drift alerts where provisioned |
| Agent-verified premium prompt packs | Not shipped in this repo | Premium/hosted content, where provisioned |
| Higher-rate endpoints and SLA | Not shipped in this repo | Premium/hosted infrastructure, where provisioned |
| Enterprise auth, audit logging, metering, and entitlement checks | Not built into `mcp_server.py` | Deployment or gateway controls, where provisioned |
| Live vendor connectors or write-path remediation tools | Not shipped in this repo | Requires separate connector or gateway implementation |

This framing keeps the open server useful and forkable while avoiding
claims that the repository does not support.
